# MEPA Operational Context System

This document defines the recommended long-term document structure for MEPA (Marshall Epie Personal Assistant).

MEPA should operate using modular context documents rather than a single massive prompt.

Reason:
- easier maintenance,
- cleaner operational separation,
- scalable memory structure,
- easier updating,
- better long-term AI orchestration.

The recommended architecture is:

```text
/mepa-context/
│
├── 00_identity_and_mission.md
├── 01_operational_behaviour.md
├── 02_front_burner_projects.md
├── 03_project_drive.md
├── 04_project_path_of_clarity.md
├── 05_tools_and_systems.md
├── 06_people_and_relationships.md
├── 07_workflows_and_processes.md
├── 08_long_term_strategy.md
└── 09_memory_and_notes.md
```

The files below are the foundational starting point.

# Recommendations

## Recommended Approach

Use MULTIPLE markdown files.

Do NOT use one giant context document.

Reason:
- easier updates,
- easier indexing,
- cleaner AI retrieval,
- modular memory,
- easier Git versioning,
- easier scaling.

---

# Recommended Workflow

## Static Files
Rarely changed:
- identity,
- philosophy,
- operational behaviour,
- tools.

## Dynamic Files
Frequently updated:
- front burner projects,
- negotiations,
- active priorities,
- temporary objectives,
- operational notes.

---

# Recommended Future Architecture

Eventually MEPA should:
- ingest markdown context automatically,
- index documents semantically,
- update memory dynamically,
- connect documents to GitHub repos,
- maintain persistent operational awareness.

This becomes the beginning of a real operational AI system rather than a simple chatbot.
